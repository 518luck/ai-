# 三层模型：AGENTS.md / Skills / Agents

AI 编程工具的规则体系可以抽象为三层，从"始终在场"到"按需派遣"：

```
┌─────────────────────────────────────────────────┐
│ 第 1 层：AGENTS.md — "宪法"（始终加载）           │
│ 回答：这个项目是什么？怎么跑？怎么写代码？         │
│ 内容：项目结构、构建命令、代码风格、架构模式         │
│ 类比：新员工入职手册                               │
├─────────────────────────────────────────────────┤
│ 第 2 层：Skills — "操作手册"（按需加载）           │
│ 回答：怎么完成某个具体任务？                       │
│ 内容：分步骤工作流、命令、代码模板、检查清单         │
│ 类比：SOP 标准操作流程                             │
├─────────────────────────────────────────────────┤
│ 第 3 层：Agents — "专家"（按需派遣）              │
│ 回答：谁来执行这个任务？                           │
│ 内容：专家身份 + 工具权限 + 引用的 skills           │
│ 类比：请了一位专业顾问                             │
└─────────────────────────────────────────────────┘
```

## 加载机制

| 层级 | 加载时机 | 存储位置 | 体积控制 |
|------|----------|----------|----------|
| AGENTS.md | 每次会话自动加载 | 仓库根目录及子目录 | 需要精简，按目录分层 |
| Skills | 触发特定任务时按需加载 | `.agents/skills/` 或 `.opencode/skills/` | 可以更详细，因为不会全部加载 |
| Agents | 需要特定能力时派遣 | `opencode.json` 中的 agents 配置 | 定义角色+权限+引用的 skills |

## 三层协作示例

以 Sentry 项目为例：

1. **AGENTS.md** 告诉 AI："本项目是 Django + React，后端代码在 `src/`，前端在 `static/app/`，测试用 pytest"
2. **Skill** (`sentry-backend-bugs`) 告诉 AI："修后端 bug 时，按这个检查清单排查：null/type errors → query validation → database integrity → concurrency bugs"
3. **Agent** 告诉 AI："你现在是一位 Sentry 后端安全专家，你有文件读写权限，使用 sentry-backend-bugs 和 sentry-security 两个 skills"
