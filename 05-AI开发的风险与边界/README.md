# 05 - AI开发的风险与边界

## 核心结论

> AI开发的本质是"用测试代替理解"。

不是纯赌博，但前提是有足够的质量保障机制。

## 三个关键前提（缺一不可）

| 前提 | 说明 | 缺失后果 |
|------|------|---------|
| 测试覆盖率 | 能自动验证代码正确性 | 无法发现AI引入的bug |
| 灰度发布 | 能逐步放量，出问题影响有限 | 一次全量上线，出问题全盘崩溃 |
| 回滚能力 | 能快速回退到已知正确版本 | 出问题后无法恢复 |

**三个都有 = 可控风险；缺少任何一个 = 赌博。**

## 不同人群的风险等级

### 资深工程师用AI

```
写自己熟悉的语言：低风险（完全理解，AI只是加速器）
写自己不熟悉的语言："半黑盒"
  - 能判断逻辑对不对（工程经验通用）
  - 不了解语言特定陷阱和惯用法
  - 不是纯赌博，但需要额外验证
```

### 完全不懂代码的人用AI

```
全黑盒：无法判断AI输出是否正确
只能看结果"能不能跑"
无法区分"能用"和"好用/安全/可维护"
= 赌博
```

## 不适合AI开发的领域

| 领域 | 原因 |
|------|------|
| 嵌入式系统 | 硬件约束严格，难以自动测试 |
| 金融核心交易 | 错误代价极高，不允许试错 |
| 医疗设备控制 | 生命安全相关，法规要求全程可追溯 |
| 航空/军事 | 安全等级要求，不能由AI生成关键代码 |

## 缺乏数据支撑的领域

- AI代码的**长期可维护性**（6个月后是否容易改）
- **技术债积累速度**（AI是否比人类产生更多技术债）
- 所有已知案例都是近1-2年内的，**缺乏长期跟踪验证**

## NAB案例的隐藏前提

> NAB支付App从零开发3周完成4个月工作量。

关键前提是**首席工程师兜底**。不是新手裸跑AI。

## Amplitude案例的隐藏前提

> 60-70%的PR由AI自动审查合并。

关键前提是**低风险PR才自动合并，高风险PR路由给工程师**。不是无差别自动合并。

## Cursor的Agent质量保障

Cursor内部如何确保Agent质量：

### 1. 测量体系

| 方法 | 说明 |
|------|------|
| CursorBench | 离线评测套件，标准化质量比较 |
| A/B测试 | 在线实验，真实用户使用 |
| Keep Rate | Agent生成的代码在一段时间后仍保留在代码库中的比例 |
| LM满意度评分 | 用另一个模型读取用户反馈判断满意度 |

### 2. 降级追踪

- 工具调用错误分类：参数错误、环境异常、Provider错误、用户中断、超时
- 未知错误 = harness bug，立即告警
- 异常检测：预期错误超过基线时触发

### 3. 模型特定优化

- 不同模型使用不同的工具格式（OpenAI用patch，Anthropic用string replacement）
- 自定义提示词适配每个provider和模型版本
- 中途切换模型时自动适配harness

## 安全实践

### 绝对不要做的事

- 不要让AI接触密钥、密码、token
- 不要把AI生成的代码直接部署到生产环境（不经测试）
- 不要在关键路径上完全依赖AI（无人工审查）

### 应该做的事

- 所有AI代码变更必须经过CI管道
- 使用Hook系统拦截对敏感文件的修改
- 保持测试覆盖率
- 保留回滚能力
