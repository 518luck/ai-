# 前端风格指南

Grafana Labs 在 React 相关事务上遵循 [Airbnb React/JSX 风格指南](https://github.com/airbnb/javascript/tree/master/react)。本指南列出了我们遵循的风格规则要点。

## 基本规则

- 尽量保持文件小而专注。
- 将大型组件拆分为子组件。
- 使用空格进行缩进。

## 命名约定

命名代码元素时请遵循以下准则。

### 类名

使用 PascalCase。例如：

```typescript
// bad
class dataLink {
  //...
}

// good
class DataLink {
  //...
}
```

### 常量

常量使用全大写。例如：

```typescript
// bad
const constantValue = "This string won't change";
// bad
const constant_value = "This string won't change";

// good
const CONSTANT_VALUE = "This string won't change";
```

### Emotion 类名

使用 camelCase。例如：

```typescript
const getStyles = (theme: GrafanaTheme2) => ({
  // bad
  ElementWrapper: css`...`,
  // bad
  ["element-wrapper"]: css`...`,

  // good
  elementWrapper: css({
    padding: theme.spacing(1, 2),
    background: theme.colors.background.secondary,
  }),
});
```

使用 `useStyles2(getStyles)` hook 来缓存样式生成，尽量避免向 `getStyles` 函数传递 props，而是使用 Emotion 的 CX 函数组合类名。

### 枚举

使用 PascalCase。例如：

```
// bad
enum buttonVariant {
  //...
}

// good
enum ButtonVariant {
  //...
}
```

### 文件和目录

根据主要导出内容命名文件：

- 当主要导出是类或 React 组件时，使用 PascalCase。
- 当主要导出是函数时，使用 camelCase。

对于导出多个工具函数的文件，使用能描述这些工具职责的名称。例如，导出数学工具的文件应命名为 `math.ts`。

- 导出常量的文件使用 `constants.ts`。
- 导出 Redux actions 的文件使用 `actions.ts`。
- Redux reducers 使用 `reducers.ts`。
- 测试文件使用 `*.test.ts(x)`。

目录名使用 dash-case（有时称为 kebab-case）。

- 例如 `features/new-important-feature/utils.ts`

### 函数

使用 camelCase。例如：

```typescript
// bad
const CalculatePercentage = () => { ... }
// bad
const calculate_percentage = () => { ... }

// good
const calculatePercentage = () => { ... }
```

### 接口

使用 PascalCase。例如：

```
// bad
interface buttonProps {
  //...
}
// bad
interface button_props {
  //...
}
// bad
interface IButtonProps {
  //...
}

// good
interface ButtonProps {
  //...
}

// bad
type requestInfo = ...
// bad
type request_info = ...

// good
type RequestInfo = ...
```

### 方法

使用 camelCase。例如：

```typescript
class DateCalculator {
  // bad
  CalculateTimeRange () {...}
}
class DateCalculator {
  // bad
  calculate_time_range () {...}
}

class DateCalculator {
  // good
  calculateTimeRange () {...}
}
```

### React 组件

遵循以下准则命名 React 组件。

#### React 回调属性和处理函数

回调属性和处理函数使用 _on_ 前缀命名。例如：

```tsx
// bad
handleChange = () => {

};

render() {
  return (
    <MyComponent changed={this.handleChange} />
  );
}

// good
onChange = () => {

};

render() {
  return (
    <MyComponent onChange={this.onChange} />
  );
}

```

#### React 组件构造函数

实现 React 组件时使用以下约定：

```typescript
// bad
constructor(props) {...}

// good
constructor(props: Props) {...}
```

#### React 组件 defaultProps

实现 React 组件时使用以下约定：

```typescript
// bad
static defaultProps = { ... }

// good
static defaultProps: Partial<Props> = { ... }
```

#### React 组件定义

实现 React 组件时使用以下约定：

```jsx
// bad
export class YourClass extends PureComponent { ... }

// good
export class YourClass extends PureComponent<{},{}> { ... }
```

#### React 状态和属性

使用 camelCase。例如：

```typescript
interface ModalState {
  // bad
  IsActive: boolean;
  // bad
  is_active: boolean;

  // good
  isActive: boolean;
}
```

### SASS

SASS 样式已弃用。在需要修改 SASS 样式时，应迁移至 Emotion。

### 类型

通常应让 TypeScript 自动推断类型，无需为每个变量显式定义类型。

但有一些例外：

```typescript
// TypeScript needs to know the type of arrays or objects; otherwise, it infers type as an array of any

// bad
const stringArray = [];

// good
const stringArray: string[] = [];
```

在新代码中显式指定函数返回类型。这样可以提高可读性，仅通过函数签名就能判断返回内容。同时也能防止函数返回类型超出作者预期范围的问题。

> **注意：** 此规则尚未启用 lint 检查，因为还有旧代码需要先修复。

```typescript
// bad
function transform(value?: string) {
  if (!value) {
    return undefined;
  }
  return applyTransform(value);
}

// good
function transform(value?: string): TransformedValue | undefined {
  if (!value) {
    return undefined;
  }
  return applyTransform(value);
}
```

### 变量

使用 camelCase。例如：

```typescript
// bad
const QueryTargets = [];
// bad
const query_targets = [];

// good
const queryTargets = [];
```

## 代码组织

将代码组织在包含功能代码的目录中：

- 将 Redux 状态和领域逻辑代码放在 `state` 目录中（例如 `features/my-feature/state/actions.ts`）。
- 将 React 组件放在 `components` 目录中（例如 `features/my-feature/components/ButtonPeopleDreamOf.tsx`）。
- 将测试文件放在被测试对象旁边。
- 将容器（页面）放在功能根目录中（例如 `features/my-feature/DashboardPage.tsx`）。
- 将非 Redux thunk 的 API 函数调用放在同目录下的 `api.ts` 文件中。
- 子组件应放在组件文件夹中。小组件不需要单独的文件夹。
- 组件的 SASS 样式应与组件代码放在同一文件夹中。

对于需要被外部插件使用的代码：

- 将组件和类型放在 `@grafana/ui` 中。
- 将数据模型和数据工具放在 `@grafana/data` 中。
- 将运行时服务接口放在 `@grafana/runtime` 中。

### 导出

- 对所有需要从文件中导出的代码使用命名导出。
- 使用声明式导出（即 `export const foo = ...`）。
- 避免使用默认导出（例如 `export default foo`）。
- 仅导出模块外部需要使用的代码。

### 代码注释

- 使用 [TSDoc](https://github.com/microsoft/tsdoc) 注释来编写代码文档。
- 使用 [react-docgen](https://github.com/reactjs/react-docgen) 注释（`/** ... */`）来编写 props 文档。
- 函数、类等内部的注释使用行内注释。
- 添加注释时请尽量遵循[代码注释指南](./code-comments.md)。

## Lint

Lint 使用 [@grafana/eslint-config](https://github.com/grafana/eslint-config-grafana) 执行。

## 函数组件

创建新的 React 函数组件时，使用函数声明而非函数表达式。例如：

```typescript
// bad
export const Component = (props: Props) => { ... }

// bad
export const Component: React.FC<Props> = (props) => { ... }

// good
export function Component(props: Props) { ... }
```

## 状态管理

- 不要在 reducers 或 thunks 中直接修改状态。
- 使用 `createSlice`。详见 [Redux Toolkit](https://redux-toolkit.js.org/)。
- 使用 `reducerTester` 测试 reducers。详见 [Redux 框架](redux.md)。
- 使用状态选择器访问状态，而非直接访问。
