### Alerting 特有约定

**使用 @grafana/alerting 包**：

- **创建新组件或 hook 前，务必先检查 @grafana/alerting 中是否已有可共享的组件或 hook**：

  ```typescript
  // 推荐 - 使用 @grafana/alerting 导出的组件/hooks
  import { AlertLabel, alertingMatchers } from "@grafana/alerting";

  // 先检查已有的实现，避免重复开发
  ```

**布局组件**：

- **优先使用 @grafana/ui 的布局组件**，而非自定义样式的 div：

  ```typescript
  // 推荐 - 使用 Box 实现简单布局/间距
  import { Box } from '@grafana/ui';
  <Box marginLeft={1}>Content</Box>

  // 推荐 - 使用 Stack 实现 flex 布局
  import { Stack } from '@grafana/ui';
  <Stack direction="column" gap={2}>
    <div>Item 1</div>
    <div>Item 2</div>
  </Stack>

  // 避免 - 当已有布局组件时，仍使用自定义样式的 div
  <div className={styles.wrapper}>Content</div>
  ```

## 状态管理模式

### RTK Query（主要方式 - 推荐）

**重要**：我们的方向是使用 RTK Query 进行数据获取，而非 Redux。

- API slices 位于 `api/` 目录
- 自定义基础查询在 `api/alertingApi.ts`
- 自动缓存，2 分钟轮询间隔：`RULE_LIST_POLL_INTERVAL_MS`
- 主要 API：`alertRuleApi`、`alertmanagerApi`、`prometheusApi`、`receiversApi`

**新功能开发**：始终使用 RTK Query hook 进行数据获取：

```typescript
import { useGetAlertRulesQuery } from "../api/alertRuleApi";

const { data, isLoading, error } = useGetAlertRulesQuery(params);
```

### 使用自动生成的 API 客户端（`@grafana/api-clients`）

**重要**：调用 API 端点时，始终优先使用 `@grafana/api-clients` 的自动生成客户端，而非手动创建 RTK Query 端点。不要手动编写新的 RTKQ 端点——请使用生成的版本。

自动生成的客户端位于 `@grafana/api-clients/rtkq/<api-group>/<version>`（例如 `@grafana/api-clients/rtkq/notifications.alerting/v0alpha1`）。

**当自动生成的客户端可直接使用时**——直接导入并使用：

```typescript
import { generatedAPI } from "@grafana/api-clients/rtkq/notifications.alerting/v0alpha1";

const { data } = generatedAPI.useListReceiversQuery(params);
```

**当自动生成的客户端不完整时**（例如缺少请求体类型），使用 `enhanceEndpoints` 覆盖端点，同时以生成的客户端为基础。这样可以避免完全手动创建 RTKQ 端点：

```typescript
import {
  CreateReceiverTestApiArg,
  generatedAPI,
} from "@grafana/api-clients/rtkq/notifications.alerting/v0alpha1";

// Define the missing body type
interface TestReceiverIntegrationBody {
  integration: { type: string; settings: Record<string, unknown> };
  alert: {
    labels: Record<string, string>;
    annotations: Record<string, string>;
  };
}

// Extend the generated arg with the correct body type
export interface CreateReceiverTestOverrideArg extends CreateReceiverTestApiArg {
  body: TestReceiverIntegrationBody;
}

// TODO: Remove this override once the auto-generated client includes the request body type
const enhancedApi = generatedAPI.enhanceEndpoints({
  endpoints: {
    createReceiverTest: (endpoint) => {
      endpoint.query = (queryArg: CreateReceiverTestOverrideArg) => ({
        url: `/receivers/${queryArg.name}/test`,
        method: "POST" as const,
        body: queryArg.body,
      });
    },
  },
});

// Re-export with correct types
export function useCreateReceiverTestMutation() {
  const [originalTrigger, state] = enhancedApi.useCreateReceiverTestMutation();
  const trigger = (arg: CreateReceiverTestOverrideArg) => originalTrigger(arg);
  return [trigger, state] as const;
}
```

**关键规则**：

1. 始终从 `@grafana/api-clients` 开始——绝不要跳过它去手动编写 RTKQ 端点
2. 如果生成的客户端存在缺陷（缺少请求体、类型错误），使用 `enhanceEndpoints` 进行修补
3. 用 `TODO` 注释标记覆盖部分，以便在生成的客户端修复后移除
4. 将增强后的 API 封装放在 `api/` 目录中（例如 `api/testReceiversApi.ts`）

import { dateTime } from "@grafana/data";

### 链接 URL 和导航

**重要**：不同的导航组件需要不同的 URL 格式。

#### 何时使用 `createRelativeUrl`

`createRelativeUrl` **仅用于 LinkButton**（以及其他渲染 HTML `<a>` 元素的组件）：

```typescript
import { createRelativeUrl } from '@grafana/data';
import { LinkButton } from '@grafana/ui';

// LinkButton renders <a> tag - needs manual subpath prefix
<LinkButton href={createRelativeUrl('/alerting/list')}>
  View Rules
</LinkButton>
```

**原因**：LinkButton 渲染的是原生 HTML 锚点元素，不使用 React Router。必须使用 `createRelativeUrl` 手动添加子路径前缀。

#### 何时不使用 `createRelativeUrl`

以下情况**不要**使用 `createRelativeUrl`：

1. **locationService** - 自动添加前缀：

```typescript
import { locationService } from "@grafana/runtime";

// locationService uses react-router history - prefix added automatically
locationService.push("/alerting/list"); // ✅ Correct - no createRelativeUrl
locationService.push(createRelativeUrl("/alerting/list")); // ❌ Wrong - double prefix!
```

2. **TextLink 组件** - 自动添加前缀：

```typescript
import { TextLink } from '@grafana/ui';

// TextLink uses react-router Link - prefix added automatically
<TextLink href="/alerting/list">View Rules</TextLink> // ✅ Correct
<TextLink href={createRelativeUrl('/alerting/list')}>View Rules</TextLink> // ❌ Wrong
```

#### 总结

| 组件/服务              | 使用 `createRelativeUrl`？ | 原因                                       |
| ---------------------- | -------------------------- | ------------------------------------------ |
| `LinkButton`           | ✅ 是                      | 渲染 `<a>` 标签（原生 HTML）               |
| 带 `href` 的 `Button`  | ✅ 是                      | 提供 href 时渲染 `<a>` 标签               |
| `locationService`      | ❌ 否                      | 使用 react-router history（自动添加前缀） |
| `TextLink`             | ❌ 否                      | 使用 react-router Link（自动添加前缀）    |
| React Router `Link`    | ❌ 否                      | React Router 组件（自动添加前缀）         |

**经验法则**：如果渲染的是原生 HTML `<a>` 标签，使用 `createRelativeUrl`。如果使用的是 React Router，则不需要。

## 依赖安全

- **未经明确批准不得引入新依赖**：不要运行 `yarn add` 或以其他方式引入新包。如果某项任务需要新依赖，先停下来向用户申请批准——说明你想要什么包、为什么需要、以及其发布日期。
- **7 天隔离期**：即使获得批准，也不要添加最新版本发布不足 7 天的依赖。在提议之前使用 `npm view <package> time --json` 检查发布日期。
- **优先使用成熟的包**：选择知名且积极维护的包。避免下载量极少或近期无维护的包。
- **禁止覆盖 postinstall 脚本**：仓库设置了 `enableScripts: false`。未经 @grafana/frontend-ops 批准，不要添加针对特定包的脚本覆盖。
- **锁定文件完整性**：始终使用 `yarn install --immutable`。永远不要手动编辑 `yarn.lock`。
- **报告可疑包**：如果依赖出现被入侵的迹象（意外脚本、混淆代码、所有权变更），在 PR 中标记并通知 @grafana/frontend-ops。
