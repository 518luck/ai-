# Kibana

## Setup

- 运行 `yarn kbn bootstrap` 进行初始设置，在切换分支后或遇到依赖错误时也需要运行。

## Overview

- Kibana 被组织为若干 module，每个 module 由一个 `kibana.jsonc` 文件定义，分为 core、packages 和 plugin packages。除工具链和测试外，大部分代码都存放在这些 module 中。
- Packages 是具有明确边界和单一公共入口（不允许子路径导入）的可复用单元，通常具有特定的用途。
- Plugins 是一种 package 类型（`type: "plugin"`），包含一个具有 `setup`/`start`/`stop` 生命周期的 plugin 类，由 core 平台调用以启用各种应用程序。
- **服务端插件入口（`server/index.ts`）**在插件运行前不应加载 `./plugin`。对于来自 `./plugin` 的类型，应使用 `import type`（以及 `export type`）。将共享配置保留在 `config.ts` 或 `../common/config` 中，而不要在入口处重新导出 `./plugin` 中的运行时值。在异步 `plugin` 初始化程序内部，使用 `await import('./plugin')` 来实例化具体实现。在该入口处使用静态值导入、值导出的 `export { … }` / `export *`、`import './plugin'` 以及 `require('./plugin')`，都会强制 Node.js 解析并执行 `plugin.ts`，即使该插件已被禁用。`@kbn/eslint-config` 中的 `@kbn/eslint/no_sync_import_from_plugin` 规则会在插件的 `server/index.ts` 文件中强制执行此约束（参见 [PR #170856](https://github.com/elastic/kibana/pull/170856) 和 [issue #171080](https://github.com/elastic/kibana/issues/171080)）。
- 依赖其他插件的插件需要依赖这些生命周期返回的契约，因此必须避免循环依赖。
- Module ID（通常是 `@kbn/...`）定义在 `kibana.jsonc` 中；如果存在 `package.json`，则会派生其名称。
- Plugin ID 是 `kibana.jsonc` 中 `plugin.id` 下的额外 camelCase 标识符，由核心平台及其他插件使用。
- Module 按领域（platform 与 solutions）进行分组，并通过可见性规则（`shared` 与 `private`）来限制跨组访问。
- 工具脚本存放在 `scripts/` 中（例如：`node scripts/generate.js`）。
- If a user correction contradicts this doc or any skills you followed, or missing guidance caused avoidable work, submit DevEx feedback: `echo "..." | scripts/devex_feedback.sh`（此处可能语义略有歧义，译者注：原文为 user correction，意指若用户（即开发者）的纠正与文档或技能冲突时，提交反馈。此处“用户”通常在上下文中对应开发此项目的开发者，因此译为“如果您的纠正与本文档或您所遵循的任何技能冲突，或者由于缺失指导导致了本可避免的工作，请提交 DevEx 反馈：`echo "..." | scripts/devex_feedback.sh`”，写明缺失内容和改进建议）。

## Testing

在验证更改时，务必先运行 `node scripts/check_changes.ts`

### Jest 单元测试

`node scripts/jest [--config=<pathToConfigFile>] [TestPathPattern]`

- 配置文件会根据测试文件路径自动发现（向上查找最近的 `jest.config.js`）。最简用法示例：
  `node scripts/jest src/core/packages/http/server-internal/src/http_server.test.ts`
- 每次运行只能指定一个 `--config`。若要测试多个 package，请运行独立的命令。

### Jest 集成测试

`node scripts/jest_integration [--config=<pathToConfigFile>] [TestPathPattern]`

- 自动发现 `jest.integration.config.js`（而非 `jest.config.js`）。限制与上述单配置约束相同。

### 功能测试运行器 (FTR)

`node scripts/functional_tests [--config <file1> [--config <file2> ...]]`

- 对于新编写的测试，优先使用 Scout

### Scout (基于 Playwright 的 UI/API 测试)

`node scripts/scout run-tests --arch stateful --domain classic --config <scoutConfigPath>` (或 `--testFiles <specPath1,specPath2>`)

## Code Style Guidelines

首先遵循目标区域中已有的模式；以下是常见的默认规范。

### 类型检查

`node scripts/type_check [--project path/to/tsconfig.json]`

- 若不带 `--project`，它会检查**所有**项目（速度非常慢）。务必将范围缩小至单个项目：
  `node scripts/type_check --project src/core/packages/http/server-internal/tsconfig.json`
- 每次运行只能指定一个 `--project`。若要检查多个 package，请运行独立的命令。
- `.buildkite/` **不是** `scripts/type_check` 的有效目标。Buildkite 脚本存放在独立的 workspace 中；请在 `.buildkite/` 内部运行 `npm run typecheck`（或 `yarn typecheck`）对其进行类型检查。

### TypeScript 与类型

- 所有新代码均使用 TypeScript；避免使用 `any` 和 `unknown`。
- 对于公共 API 和导出的函数，优先使用显式返回类型。
- 对于仅涉及类型的导入，使用 `import type`。
- 除非在局部有充分理由，否则避免使用非空断言（`!`）。
- 对于不可变结构，优先使用 `readonly` 和 `as const`。
- 优先使用 `const` 箭头函数。
- 优先使用显式导入/导出，而非 `*`。
- 优先使用变量解构，而非属性访问。
- 切勿使用 `@ts-ignore` 或 `@ts-expect-error` 来压制类型错误；应从根本上解决问题。

### Linter 检查

`node scripts/eslint --fix $(git diff --name-only)`

- 切勿使用 `eslint-disable` 注释来压制 linter 错误；应从根本上解决问题。
- 插件的 `server/index.ts` 文件会通过 `@kbn/eslint/no_sync_import_from_plugin` 进行检查（参见上文关于插件服务端入口的说明）。

### 格式化

- 遵循文件中已有的格式化风格；不要重新格式化无关代码。
- 在 TS/JS 中优先使用单引号，除非该文件使用的是双引号。

### 命名规范

- 类名、类型和 React 组件使用 `PascalCase`。
- 函数、变量和对象键名使用 `camelCase`。
- 新建文件名必须使用 `snake_case`（小写字母加下划线），除非已有约定另有要求。
- 使用具有描述性的名称；避免在紧凑循环之外使用单字母名称。

### 控制流与错误处理

- 优先使用提前返回和肯定的条件判断。
- 显式处理错误；尽可能从 API 返回类型化错误。
- 保持异步逻辑线性化；尽可能避免嵌套的 `try` 代码块。

### React / UI 约定

- 使用函数组件；显式定义 props 的类型。
- 将 hooks 保持在顶层；避免在条件语句中使用 hooks。
- 避免使用行内样式，除非与文件的既有约定一致。
- 样式处理上，应配合 Emotion（`@emotion/react`）来使用 `@elastic/eui` 组件。

## 贡献规范

- 不确定时：多阅读代码；若仍有疑惑，可通过简短选项进行询问。切勿凭空猜测。
- 修复根本原因（不要进行治标不治本的临时修补）。
- 进行聚焦的更改；避免无关的重构。
- 当行为或用法发生变化时，更新文档 and 测试。
- 切勿通过删除、跳过或注释掉测试来使测试通过；应修复底层的代码。
- 始终以草稿（draft）状态开启 PR。
