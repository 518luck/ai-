## 基础 Prompt 的发送机制

### 什么时候基础 Prompt 会被发送

真正发送逻辑在：[packages/opencode/src/session/llm/request.ts:58](file:///home/duoyun/idea/open-source/opencode-v1.15.10/packages/opencode/src/session/llm/request.ts#L58)

```typescript
...(input.agent.prompt ? [input.agent.prompt] : SystemPrompt.provider(input.model)),
```

**意思是：**

- 如果当前 agent 有自己的 prompt，就用 agent 自己的 prompt。
- 如果当前 agent 没有自己的 prompt，才按模型选择这些基础 prompt。
- `build`、`plan`、`general` 默认没有专属 prompt，所以会走这里。
- `explore`、`scout`、`compaction`、`title`、`summary` 有专属 prompt，所以默认不会用这些基础 prompt。

### 每次请求都会发吗

**是的**，每次 LLM 请求都会重新构造 system 信息。

如果当前是 `build`，当前模型命中 `default.txt`，那么每次请求都会带上：

- `default.txt`
- 环境信息
- `AGENTS.md` / `instructions`
- `skills` 信息
- 当前会话历史
- 可能的 `synthetic reminder`

环境信息来自：[packages/opencode/src/session/system.ts:48](file:///home/duoyun/idea/open-source/opencode-v1.15.10/packages/opencode/src/session/system.ts#L48)

包含：

- `model id`
- `working directory`
- `workspace root`
- `git repo` 状态
- `platform`
- `date`

## TUI / Web / CLI 是否变化

基础 prompt 选择本身不看 TUI/Web。它主要取决于：

- 当前 agent 是否有自己的 prompt
- 当前 `model.api.id`
- 当前 `runtime flags`
- 当前会话状态

所以同一个 agent、同一个模型下：

- TUI -> 同一套基础 prompt
- Web -> 同一套基础 prompt
- CLI run -> 同一套基础 prompt
- Desktop -> 同一套基础 prompt

但有一个重要例外：
`plan_exit` 工具只在这里被加入：[packages/opencode/src/tool/registry.ts:271](file:///home/duoyun/idea/open-source/opencode-v1.15.10/packages/opencode/src/tool/registry.ts#L271)

```typescript
...(flags.experimentalPlanMode && flags.client === "cli" ? [tool.plan] : [])
```

所以实验版 plan mode 的 `plan_exit` 工具只在 `flags.client === "cli"` 时启用。`OPENCODE_CLIENT` 如果是 desktop 或 acp，这个工具不会加入。基础 prompt 文件不变，但可用工具会变。

## 最终发送结构

普通 build + 无专属 prompt 的请求，大概是：

- 基础 prompt：`default`/`gpt`/`anthropic`/`gemini`/...
- 环境信息：工作目录、git、平台、日期
- instructions：`AGENTS.md` 等
- `skills` 信息
- 用户消息历史
- 工具说明

如果当前是 plan，还会追加：

- `plan.txt` 或 `plan-mode.txt`

如果刚从 plan 切回 build，还会追加：

- `build-switch.txt`

如果达到最大步骤数，还会追加：

- `max-steps.txt`

**核心结论**：这些文件不是都一起发。每次基础 prompt 只选一个；Plan、Build switch、Max steps、Reference 是根据会话状态额外注入。
