### 🔧 代码示例

> OpenCode 实践证明：Good/Bad 对比示例对 AI 理解风格偏好极为有效。

```ts
// ✅ Good — [简要说明为什么好]
const result = condition ? doA() : doB()

// ❌ Bad — [简要说明为什么不好]
let result
if (condition) {
  result = doA()
} else {
  result = doB()
}
```

```ts
// ✅ Good — 减少变量数量，直接内联
return items.filter(isActive).map(toResponse)

// ❌ Bad — 不必要的中间变量
const activeItems = items.filter(isActive)
const responses = activeItems.map(toResponse)
return responses
```

## AI 行为指令

> 以下指令指导 AI 代理的工作方式，而非代码风格。

- 优先自动化执行，除非操作不可逆或信息不足
- 尽可能使用并行工具调用（ALWAYS USE PARALLEL TOOLS WHEN APPLICABLE）
- 不确定时：先阅读相关代码；仍不确定则提问，绝不猜测
- 修复根因而非表面修补（Fix root cause, not band-aid）
- 聚焦变更，避免无关重构
- 行为或接口变更时，同步更新文档和测试
- 🔧 先读目标区域的现有代码，遵循已有模式（Follow existing patterns）

## 模板使用说明

### 必填项（所有项目都应包含）

| 章节 | 理由 |
|------|------|
| **元信息** | 让 AI 知道基本项目参数 |
| **项目概览** | 帮助 AI 理解项目上下文 |
| **常用命令速查** | 测试、lint、类型检查是 AI 最常执行的操作 |
| **代码风格指南** | 通用原则至少需要 Always/Never 规则 |
| **提交与 PR 规范** | 确保提交质量一致性 |

### 按需选填项（根据项目特点选用）

| 章节 | 适用场景 |
|------|----------|
| **上下文感知引导** | 大型 monorepo、前后端分离项目 |
| **代码示例** | 有强烈风格偏好的项目（强烈推荐） |
| **AI 行为指令** | 需要优化 AI 工作效率的项目（强烈推荐） |
| **构建命令** | 构建步骤非平凡的项目 |
| **项目特定规则** | 有架构约束、安全要求、Feature Flags 等 |
| **国际化** | 有多语言支持需求的项目 |

### 行数控制建议

| 项目规模 | 建议行数 | 策略 |
|----------|----------|------|
| 小型项目（< 1000 文件） | 50-100 行 | 单文件，只保留必填项 |
| 中型项目（1000-10000 文件） | 100-200 行 | 单文件，适当添加选填项 |
| 大型项目（> 10000 文件） | 100-150 行根文件 + 子目录文件 | 利用上下文感知引导，根文件只放概览和通用规则 |

### 从参考文件中学到的关键技巧

1. **Kibana**：每个命令都带完整路径和参数说明，避免模糊指令
2. **OpenCode**：Good/Bad 代码对比示例比文字描述有效 10 倍
3. **Sentry**：层级式 AGENTS.md 是大型项目的最佳实践
4. **WP Calypso**：Pre-PR 检查清单能显著减少 CI 失败率
