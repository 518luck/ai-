# AI coding

- AI coding agent 虽然可以根据业务描述,代码里面的下文、AGENTS.md、skills、类型检查和测试命令，完成大部分真实开发工作。但它不是“输入需求后自动生成正确架构的编译器”，更像是一个能读代码、改代码、跑检查、迭代修复的开发协作者。
- AGENTS.md 和 skills 是指导，不是强约束。它们能提高一致性，但不能保证 AI 一定遵守。真正的硬约束需要 typecheck、lint、tests、CI、hooks、code review。

  > - “生成符合规范、标准的项目架构”不能默认成立。需求越模糊，AI 越会用猜测补全，比如数据源、权限、分页、错误状态、空状态、URL 参数、服务端/客户端边界。
  > - AI 编码代理 = 一个能读取项目规范、参考现有代码、实现功能、运行检查并修复部分问题的自动化开发助手。
  > - AGENTS.md / skills = 长期项目说明和流程提示。

## AI开发流程

应该的AI开发流程为

**将"业务需求" =(转换为)=> 工程任务**

### 示例

接收到的业务 : 加一个查询按钮现在用户只能一个个查找,太麻烦了。

工程任务 ：

- 在 Next.js App Router 的 /xxx 页面添加查询功能。
- 查询字段是 status，值来自 URL searchParams。
- 服务端根据 searchParams 查询数据，结果展示到现有 shadcn table。
- 查询表单使用 shadcn Button 和 Input。
- 先阅读现有页面、数据访问层、表格组件模式，再实现。
- 完成后运行 typecheck、lint 和相关测试。

## 工程任务的编写

1. 业务边界（产品逻辑）
2. 核心实现链路（技术方案）
   > AI总是给出常用且最简单的方案,但是实际上很多公司已经有自己的体系,比如身份体系,数据体系,服务边界,查询链路等等,所以说清楚实现的具体细节
   > 但是不要指定太多技术细节,否则这会限制 AI 复用项目已有模式。
3. 验收标准表达

## 黑盒开发的后果

- 自身越来越依赖 AI，原地踏步。
- 丧失对项目的掌控力,只能依靠测试结果来反馈对错,但是无法理解代码。风险会从开发阶段后移到测试、联调甚至线上。
- 后续排查问题时，不知道从页面、接口、权限、数据源还是状态流转入手。
- 无法有效复用已有成熟方案，反而需要大量时间反复微调。
- AI 写出的代码可能能跑，但不符合公司长期架构。
