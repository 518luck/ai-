# 普通大型项目如何约束 AI 开发

## 核心问题

我想要找到一个“度”：让 AI 能参与开发，但生成的代码不会偏离项目太狠。

关注点不是“AI 能不能写代码”，而是成熟项目如何用规则、测试、review 和目录边界限制 AI 的发挥范围。

## 指导思想

AI 更适合做受控执行者，而不是完全替代架构决策者。

成熟项目通常不会主要让 AI 做这些事：

- 从 0 设计一个大功能
- 重构核心架构
- 决定目录结构
- 设计重大 API
- 绕过现有测试和 review 流程直接合并代码

更合理的使用方式是：

- 在已有架构里实现局部功能
- 按现有模式补测试
- 修 lint/typecheck/test 失败
- 按目录规则修改代码
- 按专项 skill 处理迁移、API、E2E、review 等任务
- 由人类保留架构、产品和合并决策权

## 筛选标准

优先研究这些项目：

- 产品本身不是 AI 开发工具
- 不是 AI 文档或规则展示项目
- 是真实业务软件、基础设施产品或平台型项目
- 仓库公开提交了 `AGENTS.md`、`.agents/skills`、`.claude`、`.cursor`、`.codex` 等规则文件
- 规则内容约束真实开发流程：测试、迁移、API、前端、PR、review、安全、CI

排除或降权这些项目：

- AI 原生工具项目，例如 `google-gemini/gemini-cli`、`openai/codex`、`continuedev/continue`
- AI 应用/工作流平台，例如 `Dify`
- 主要是 AI 规则或文档示例的项目

## 优先深入对象

建议优先深入这三个：

1. `getsentry/sentry`
2. `elastic/kibana`
3. `grafana/grafana`

它们更接近“真实普通大型项目公开 AI 开发规则”的目标。

## 1. `getsentry/sentry`

普通产品：错误追踪、性能监控平台。

主要开发语言：Python、TypeScript。

技术栈大致是：

- 后端：Python、Django、Django REST Framework、Celery
- 前端：TypeScript、React
- 数据和基础设施：PostgreSQL、Redis、ClickHouse、Kafka

公开内容：

- `AGENTS.md`
- `src/AGENTS.md`
- `static/AGENTS.md`
- `tests/AGENTS.md`
- `.agents/skills/*`

适合学习的点：

- 大型 Django + React 项目如何约束 AI
- 后端、前端、测试分别用目录级 `AGENTS.md` 约束
- API endpoint 规则
- Django model / migration 规则
- frontend design system 规则
- security review 规则
- lint fix 规则
- snapshot tests 规则
- hybrid cloud 架构边界

值得重点看：

- 根 `AGENTS.md`：全局开发命令、PR、分区规则
- `src/AGENTS.md`：后端开发、安全、API、数据库、性能规则
- `static/AGENTS.md`：前端组件、API 请求、测试、设计系统规则
- `.agents/skills/`：专项任务，如 migration、design system、security、lint、snapshot tests

评价：

这是目前最值得深入的样本之一。它不是 AI 产品，但公开了非常多 AI agent 规则，而且规则直接约束真实业务代码。

## 2. `elastic/kibana`

普通产品：Elastic/Kibana UI 平台。

主要开发语言：TypeScript。

技术栈大致是：

- 前端和平台层：TypeScript、React、Elastic UI
- 服务端和工具链：Node.js、TypeScript
- 测试：Jest、Scout、Functional Test Runner

公开内容：

- `AGENTS.md`
- `.agents/skills/*`
- `.claude/`
- `.codex/`

适合学习的点：

- 大型 TypeScript monorepo 如何限制 AI 修改边界
- plugin/package/module 边界规则
- API authorization 规则
- OpenAPI/OAS 校验规则
- i18n 规则
- accessibility 规则
- bundle size 优化规则
- UI/API 自动化测试规则
- branch readiness checks

公开 skills 很工程化：

- `api-authz`
- `accessibility`
- `kibana-api`
- `kibana-i18n`
- `scout-ui-testing`
- `scout-api-testing`
- `validate-oas`
- `optimize-bundle-size`
- `branch-readiness-checks`

评价：

这是研究“超大型前端/平台项目如何给 AI 设置专项工作流”的好样本。它的规则重点不是让 AI 自由写功能，而是让 AI 在严格模块边界、测试工具和 CI 规则下工作。

## 3. `grafana/grafana`

普通产品：监控和可观测性平台。

主要开发语言：Go、TypeScript。

技术栈大致是：

- 后端：Go
- 前端：TypeScript、React
- 包管理和构建：Yarn workspaces、Go workspaces
- 代码生成：Wire DI、CUE schemas、OpenAPI/Swagger、feature toggles

公开内容：

- `AGENTS.md`
- `docs/AGENTS.md`
- `public/app/features/alerting/unified/AGENTS.md`

适合学习的点：

- 明确要求 human review gate：push 前必须等人批准
- frontend/backend 分开 PR
- Go / React / codegen / feature toggles / migrations 都有具体命令
- 后端服务层、API handler、数据库迁移的边界清楚
- 前端 feature 目录、Redux Toolkit、RTK Query、Emotion CSS-in-JS 有明确约束

评价：

Grafana 的规则最能体现“AI 作为受控工程助手”。它没有过多复杂的 agent 体系，但它明确要求测试、代码生成、人类 review 和 PR 边界。

## 4. `home-assistant/core`

普通产品：智能家居平台。

主要开发语言：Python。

技术栈大致是：

- 后端核心：Python
- 测试：pytest、Syrupy snapshot testing
- 包管理和运行：uv、venv

公开内容：

- `AGENTS.md`
- `.agent/`
- `.claude/`
- `.gemini/`

适合学习的点：

- Python 项目如何约束 AI 写集成
- 测试函数参数必须有类型标注
- 避免 `Any`，优先具体类型
- integration quality scale 作为参考标准
- 不要对已经由 schema 验证过的输入做多余防御式检查
- PR template 不能删内容

评价：

它不像 Sentry/Kibana 那么复杂，但“非 AI 产品”属性最干净。适合研究一个大型 Python 开源项目如何用简单、直接的规则约束 AI。

## 5. `Automattic/wp-calypso`

普通产品：WordPress.com / Jetpack dashboard。

主要开发语言：TypeScript、JavaScript。

技术栈大致是：

- 前端：TypeScript、JavaScript、React、Redux、TanStack Query、TanStack Router
- 测试：Jest、Playwright
- 包管理：Yarn

公开内容：

- `AGENTS.md`
- `CLAUDE.md`
- `.claude/agents`
- `.claude/rules`
- `.claude/skills`
- `.codex/`
- `.cursor/`
- `test/e2e/AGENTS.md`

适合学习的点：

- 复杂前端 SaaS dashboard 如何给 AI 写规则
- Calypso、Jetpack Cloud、Dashboard 等多个 client 的边界
- PR 创建和描述规则
- typecheck-client 作为 PR 前置检查
- E2E 测试迁移规则
- 新旧 Playwright 测试框架如何区分

术语补充：

- E2E 是 `End-to-End`，即端到端测试
- 它会模拟真实用户，从打开页面、登录、点击、提交，到最终结果检查整条链路
- 常见工具是 Playwright、Cypress、Selenium

评价：

有一点 AI 功能模块，但主产品不是 AI。适合研究复杂前端产品如何约束 AI 写测试、跑 typecheck 和处理 PR。

## 6. `PostHog/posthog`

边缘但仍比 Dify 更接近普通产品：产品分析平台，但现在也有 LLM analytics / MCP / AI 相关模块。

主要开发语言：Python、TypeScript。

技术栈大致是：

- 后端：Python、Django、Django REST Framework、Celery
- 前端：TypeScript、React、Kea
- 数据：ClickHouse、PostgreSQL
- 测试：pytest、Jest、Playwright

公开内容：

- `AGENTS.md`
- `AI_POLICY.md`
- `.agents/security.md`
- `.agents/skills/*`

适合学习的点：

- tenant isolation 规则非常具体
- ClickHouse migration 规则
- Django migration 规则
- generated API types 规则
- DRF endpoint 规则
- Playwright 测试规则
- CI debugging 规则
- security / SQL / HogQL / semgrep 指南
- 明确说明什么时候应该用 linters、lint-staged、skills、AGENTS.md

强项：

- 工程规则非常细
- skills 覆盖真实开发任务
- PR 描述有公开仓库安全要求，不能泄露内部系统、客户数据、私有事故信息

不足：

- 项目里已经有 LLM analytics、MCP 等 AI 相关模块
- 仍然比纯普通产品更接近 AI 方向

评价：

可以作为补充样本。它的规则质量很高，但不是最干净的“非 AI 项目”。

## 7. `langgenius/dify`

边缘样本：AI workflow 平台。

主要开发语言：Python、TypeScript。

技术栈大致是：

- 后端：Python
- 前端：TypeScript、React / Next.js
- 测试：Cucumber、Playwright

公开内容很系统：

- `AGENTS.md`
- `api/AGENTS.md`
- `web/AGENTS.md`
- `e2e/AGENTS.md`
- `.agents/skills/*`

适合学习的点：

- 根规则把项目拆成 backend、frontend、docker、agent backend
- backend 规则非常细
- DDD / Clean Architecture
- controller -> service -> core/domain
- SQLAlchemy session 模式
- `tenant_id` 必须贯穿共享资源访问
- 禁止直接读环境变量，使用 `configs.dify_config`
- E2E 规则完整
- Cucumber + Playwright
- `DifyWorld`
- tags：`@authenticated`、`@unauthenticated`、`@fresh`
- 失败自动截图和 HTML dump

skills 覆盖：

- `backend-code-review`
- `frontend-code-review`
- `frontend-testing`
- `component-refactoring`
- `e2e-cucumber-playwright`

为什么放最后：

- 它的规则体系完整，但产品本身就是 AI workflow
- 和普通业务软件仍有距离
- 可以作为“全栈 AI 产品如何写规则”的补充参考
- 不适合作为主要学习对象

评价：

适合学习“AI 如何在大型全栈项目里遵守分层架构和测试体系”，但不适合作为普通项目 AI 开发流程的主样本。

## 最终建议

优先深入：

1. `getsentry/sentry`
2. `elastic/kibana`
3. `grafana/grafana`

补充参考：

1. `home-assistant/core`
2. `Automattic/wp-calypso`
3. `PostHog/posthog`

最后作为对照：

1. `langgenius/dify`
